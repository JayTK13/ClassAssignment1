Flutter Team Challenge: Build-A-Feature Collaborative Coding Sprint

üìã Overview

This hands-on activity transforms the foundational Flutter & Dart skills from In-Class 00 into a collaborative team-building experience using GitHub. Students will work in teams to extend their basic Flutter app through strategic feature additions, code reviews, and merge conflict resolution‚Äîmirroring real-world development workflows.

üéØ Learning Objectives

By completing this activity, students will:

  • ‚úÖ Practice Git branching, committing, pushing, and pull requests
  • ‚úÖ Experience collaborative code reviews and constructive feedback
  • ‚úÖ Resolve merge conflicts in a controlled environment
  • ‚úÖ Apply Flutter/Dart concepts: widgets, styling, and UI components
  • ‚úÖ Develop teamwork and communication skills in software development
  • ‚úÖ Build confidence through incremental feature implementation

üë• Team Structure (Teams of 3-4 students)

Role Rotation System (roles rotate every 30 minutes):
1. Feature Developer - Implements assigned features
2. Code Reviewer - Reviews PRs and provides feedback
3. Integration Manager - Manages merges and resolves conflicts
4. Documentation Lead - Updates README and comments code

üõ†Ô∏è Setup Phase (15 minutes)

Step 1: Repository Initialization

One team member (designated as Repo Owner) creates the base repository:

# Create new Flutter project
flutter create team_app_[team_name]
cd team_app_[team_name]

# Initialize git and create repository
git init
git add .
git commit -m "Initial commit: Flutter starter project"

# Create GitHub repository and push
# (Use GitHub CLI or web interface)
gh repo create team_app_[team_name] --public --source=. --push

Step 2: Team Member Setup

All other team members:

# Clone the repository
git clone https://github.com/[repo-owner]/team_app_[team_name].git
cd team_app_[team_name]

# Verify setup
flutter doctor
flutter pub get

Step 3: Replace Starter Code

Repo Owner replaces lib/main.dart with the starter code from the guide, commits, and pushes:

# After replacing code in main.dart
git add lib/main.dart
git commit -m "Add starter code with 8 customization tasks"
git push origin main

All team members pull the latest changes:

git pull origin main

üèóÔ∏è Feature Development Rounds (3 rounds √ó 30 minutes each)

Each round follows this workflow:

Round Structure:

Phase 1: Branch & Develop (15 min)

Each Feature Developer creates a branch and implements their assigned feature:

# Create and switch to feature branch
git checkout -b feature/[feature-name]

# Make changes, test with hot reload
flutter run

# Commit changes
git add .
git commit -m "Add [feature description]"
git push origin feature/[feature-name]

Phase 2: Pull Request & Review (10 min)

  1. Developer creates a Pull Request on GitHub
  2. Code Reviewer examines the code and:
    • Tests the feature locally
    • Checks code quality and comments
    • Requests changes or approves
  3. Developer addresses feedback if needed

Phase 3: Merge & Integrate (5 min)

Integration Manager merges approved PRs:

  • Resolves any merge conflicts
  • Ensures the app still runs correctly
  • Communicates status to the team

üéØ Merge Conflict Simulation (Controlled Learning)

‚ö†Ô∏è Instructor-Guided Activity (15 minutes)

This controlled exercise helps students learn conflict resolution in a safe environment.

Scenario Setup:

Two students simultaneously modify the same section of code:

  • Student A: Changes the AppBar title to "Team Alpha App"
  • Student B: Changes the AppBar title to "Flutter Champions"

Resolution Steps:

  1. Both students create branches and push their changes
  2. First PR gets merged successfully
  3. Second student encounters merge conflict when pushing
  4. Integration Manager helps resolve:
# Pull latest changes
git pull origin main

# Git indicates conflict in lib/main.dart
# Open file and choose/combine changes between <<<< and >>>>

# After resolving:
git add lib/main.dart
git commit -m "Resolve merge conflict in AppBar title"
git push origin feature/[branch-name]

üåü Why This Activity Works

  1. Scaffolded Complexity: Progresses from simple text changes to complex widgets
  2. Real-World Simulation: Mirrors actual development team workflows
  3. Immediate Feedback: Hot reload provides instant visual confirmation
  4. Collaborative Learning: Students learn from peers' code during reviews
  5. Safe Failure Environment: Conflicts are learning opportunities, not disasters
  6. Transferable Skills: Git/GitHub skills apply beyond Flutter development

üìù Instructor Preparation Checklist

  • Create example repository demonstrating completed features
  • Prepare troubleshooting guide for common Git errors
  • Set up GitHub Organization for easy repository management
  • Test all feature code examples in advance
  • Prepare merge conflict demonstration materials
  • Create grading rubric template in GitHub Classroom (optional)
  • Have backup plan for students with setup issues (pair programming)

üöÄ Getting Started for Instructors

Pre-Class Setup:

  1. Share this activity guide 1-2 days before class
  2. Have students verify Flutter installation beforehand
  3. Create team assignments and share on LMS
  4. Set up GitHub Organization (optional but recommended)

Day-of Setup:

  1. Quick Git/GitHub refresher (10 min)
  2. Demonstrate creating a branch and PR (5 min)
  3. Show expected final result from example repo (5 min)
  4. Begin team activities!
